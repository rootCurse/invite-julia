# Технологический стек — сайт-приглашение Александр & Юлия

Выбранное направление: **простой статический сайт**, минимум затрат на поддержку, полный адаптив, запас по времени — несколько месяцев до рассылки.

## 1. Обзор решения

Одностраничник без бэкенда и базы данных. Вся «динамика» (таймер, анимация конверта, карта, RSVP) реализуется на клиенте + один бессерверный endpoint для приёма RSVP в Google Таблицу. Хостинг — статический, с автодеплоем из Git.

## 2. Рекомендуемый стек

| Слой | Технология | Почему |
|---|---|---|
| Сборка/dev-сервер | **Vite** | Быстрый dev-сервер с HMR — удобно показывать правки в браузере вживую по ходу работы. `vite build` даёт чистый статический бандл (HTML/CSS/JS) без лишнего фреймворка — секции пишутся как обычные JS-модули (по одному на блок: `cover.js`, `countdown.js`, `rsvp.js`…), которые рендерят/оживляют свою разметку. Компактно для одностраничника, легко править блоки (адреса/программа появятся позже). |
| Стили | CSS с кастомными свойствами (design tokens) | Палитра всего из 2 цветов (чёрный/белый) — Tailwind избыточен, чистые CSS-переменные проще поддерживать и совпадают с токенами из Figma |
| Анимации | CSS keyframes/transitions + Intersection Observer для скролл-реveal; при необходимости лёгкая либа **motion one** (~4кб) для анимации конверта | Не тянуть тяжёлый GSAP ради одной анимации открытия конверта |
| Таймер обратного отсчёта | Vanilla JS (`Date`), без библиотек | Тривиальная логика, не нужна зависимость |
| Карта проезда | Встроенный iframe **Яндекс.Карт** (конструктор виджетов, без API-ключа) | Для RU-аудитории Яндекс.Карты привычнее Google Maps; iframe — без JS SDK и лимитов API |
| Шрифты | Google Fonts: **Yeseva One**, **Montserrat**, **Marck Script** — самохостинг через `@fontsource/*` | Не грузить сторонний Google Fonts CDN на каждый визит — быстрее и без внешнего запроса |
| RSVP-форма | HTML-форма → **Google Apps Script Web App** (doPost) → запись строки в Google Таблицу | Сохраняет кастомный дизайн формы (чёрно-белый, в стиле сайта) в отличие от вставки готовой Google Форм; не требует бэкенда/БД |
| Хостинг | **Cloudflare Pages** (альтернативы: Vercel, Netlify) | Бесплатный тариф с запасом, автодеплой из Git по пушу, бесплатный SSL, привязка домена в 2 клика |
| Домен | Регистратор reg.ru/nic.ru (если `.ru`) или Namecheap (если `.com`) | См. `domain-options.md` |
| Аналитика (опц.) | Яндекс.Метрика или Plausible | Чтобы видеть, сколько гостей открыли ссылку/дошли до RSVP |

## 3. Альтернатива «ещё проще» (если не нужен даже Vite)

Один HTML-файл + CSS + vanilla JS, без сборки и без Node.js в проекте. Деплой — просто перетащить папку в Netlify Drop или запушить в GitHub Pages. Подходит, если не планируется частое обновление контента после запуска. Минус — сложнее поддерживать по мере роста количества блоков (13 секций + анкета + доп. блоки) без модульности и HMR, поэтому для этого проекта **рекомендуется Vite**.

## 4. RSVP → Google Таблица: как это работает

1. В Google Sheets создаётся таблица с колонками, соответствующими полям формы (см. `content-structure.md`, блок 9).
2. В связанном Google Apps Script пишется функция `doPost(e)`, которая парсит тело запроса и добавляет строку через `SheetsApp.appendRow(...)`.
3. Скрипт публикуется как Web App (доступ — «Anyone», без авторизации гостя).
4. Фронтенд отправляет `fetch(APPS_SCRIPT_URL, { method: 'POST', body: JSON.stringify(formData) })` при сабмите формы, показывает статус «Спасибо, ответ получен» без перезагрузки страницы.
5. Простая защита от спама: honeypot-поле + троттлинг на уровне Apps Script (проверка timestamp последней отправки с того же IP/сессии не нужна — трафик на свадебный сайт закрытый, по прямой ссылке).

Тот же механизм используется для анкеты гостей (блок 10) — либо отдельный лист в той же таблице, либо доп. колонки.

## 5. Производительность и SEO

- Изображения — плейсхолдеры на этапе макета, в проде: заранее подготовленные WebP/AVIF (или плагин `vite-plugin-image-optimizer` в сборке), `loading="lazy"` на всех фото ниже первого экрана.
- Первый экран (обложка) оптимизируется под LCP < 2.5с на 4G.
- OpenGraph-теги (og:title, og:image) — чтобы ссылка красиво разворачивалась при отправке в WhatsApp/Telegram.
- `favicon` + монограмма как og:image.
- SEO минимален по смыслу (приглашение не для поисковой выдачи), но базовые мета-теги и `robots.txt` с `noindex` — чтобы сайт не индексировался публично (приватность гостей и деталей свадьбы).

## 6. Структура репозитория (примерная)

```
/
├── index.html            # разметка всех секций по порядку
├── src/
│   ├── sections/          # cover.js, countdown.js, timeline.js, rsvp.js, mapBlock.js, ...
│   ├── styles/            # tokens.css (цвета, шрифты, отступы) + styles per section
│   └── main.js            # точка входа, импортирует и инициализирует секции
├── public/
│   └── images/            # плейсхолдеры → реальные фото позже
├── vite.config.js
└── package.json
```

## 7. Деплой-пайплайн

1. Git-репозиторий (GitHub/GitLab).
2. Push в `main` → Cloudflare Pages запускает `npm run build` (`vite build`, вывод в `dist/`) → деплой на прод-домен.
3. Preview-деплои на каждый PR/ветку — удобно показывать промежуточные версии заказчику до финального запуска.

## 8. Ориентировочный таймлайн

Учитывая запас в несколько месяцев до рассылки приглашений:

| Этап | Срок |
|---|---|
| Макеты (Figma) | эта итерация |
| Вёрстка статики + базовые блоки | 1–2 недели |
| Интеграция RSVP (Apps Script) + анкета | 2–3 дня |
| Наполнение реальным контентом (фото, адреса, программа) | по мере получения данных от заказчика |
| Тестирование на устройствах, финальный QA | 2–3 дня |
| Запуск + рассылка ссылки гостям | за 1–2 месяца до 17.10.2026 |
